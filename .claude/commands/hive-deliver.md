# /hive-deliver

Run the **Deliver Hub** in isolation. Transforms the validated composition into a concrete, actionable artifact — code, specification, documentation, or a dev ticket.

## Usage

```
/hive-deliver
/hive-deliver --load-state <path to state file>
/hive-deliver --format code
/hive-deliver --format markdown
/hive-deliver --format json
/hive-deliver --format ticket
/hive-deliver --format all
```

## Arguments

- `--load-state` (optional): Load an existing state file; Deliver reads from `state.compose` and `state.verify`
- `--format` (optional): Output format. Options: `code` (default), `markdown`, `json`, `ticket`, `all`

---

## Hub Purpose

The Deliver Hub is the **execution layer**. It takes the verified, validated blueprint and turns it into something a developer, designer, or product person can immediately use.

Deliver Hub does not create new decisions — it faithfully renders the decisions made in Compose and confirmed in Verify. It is a **transformation step**, not a design step.

---

## Entry Conditions

Before running, verify:
- `state.verify.valid = true`
- `state.compose.blueprint` is non-empty

**If `state.verify.valid = false` or Verify has not been run, HALT immediately.** Do not produce any output. Tell the user: "Delivery is blocked — the composition has not passed verification. Run `/hive-verify` first."

This is a hard rule. It cannot be bypassed.

## Agents to Invoke

### 1. Code Generator Agent

**Mission**: Produce a code scaffold from the verified blueprint.

**Skills**: `generate_code_scaffold`

**Rules**:
- Use the framework and language from `state.ground.platform`
- Generate real, runnable scaffold code — not pseudocode
- Apply tokens from `state.ground.tokens` as CSS custom properties, Tailwind classes, or styled-components, according to the project's CSS approach
- Each component in `state.compose.components` becomes an instantiated element in the scaffold
- Include TODO comments for parts that require developer implementation (e.g., API call for payment submission)
- Follow naming conventions from `.hive/rules/naming.md` if present
- Accessibility attributes identified in Verify must be present in the code

**Output**: A code string or set of files (component file, optionally a styles file, optionally a types file)

**Example output** (React):
```tsx
// CheckoutPage.tsx — generated by HIVE Deliver Hub
// Blueprint: checkout-page | Run ID: <run_id>

import { Button } from '@ds/Button';
import { Input } from '@ds/Input';
import { Card } from '@ds/Card';
import { CartSummary } from './CartSummary';

export function CheckoutPage() {
  return (
    <div className="checkout-layout" style={{ display: 'grid', gridTemplateColumns: '40% 60%' }}>
      <section aria-label="Cart summary">
        <CartSummary />
      </section>
      <section aria-label="Payment details">
        <Card>
          <form onSubmit={/* TODO: implement payment submission */}>
            <label htmlFor="card-number">Card number</label>
            <Input id="card-number" type="text" placeholder="1234 5678 9012 3456" />
            {/* TODO: Add expiry, CVV fields */}
            <Button type="submit" variant="primary">Pay now</Button>
          </form>
        </Card>
      </section>
    </div>
  );
}
```

### 2. Specification Writer Agent

**Mission**: Produce a structured, human-readable specification document from the blueprint.

**Skills**: `generate_markdown_spec`

**Rules**:
- The specification must cover: overview, layout structure, component list, interaction flows, data model, accessibility notes
- Write for a developer or technical designer reading it for the first time
- Include the success criteria from `state.intent.success_criteria` and note which ones are met
- Note any `grounding_gaps` from `state.compose.blueprint.metadata.grounding_gaps`

**Output**: A markdown document

### 3. Documentation Agent

**Mission**: Generate user-facing or team-facing documentation for the delivered artifact.

**Skills**: `generate_documentation`

**Rules**:
- Audience: developers who will maintain or extend this component/page
- Include: purpose, usage example, props/API, accessibility notes, related components
- Use the project's existing documentation format if detectable from `.hive/prompts/conventions.md`

**Output**: A markdown file suitable for a docs site or README

### 4. Delivery Formatter Agent

**Mission**: Package and format the final output according to the requested format and project conventions.

**Skills**: `write_output_files`

**Rules**:
- Match the format to `--format` argument or `hive.config.json` `output_formats`
- For `all`: produce code + specification + documentation
- For `ticket`: produce a structured dev ticket with title, description, acceptance criteria, technical notes, linked components
- If writing to files, suggest file paths consistent with the project structure

**Ticket format** (`--format ticket`):
```markdown
## [TICKET] Create Checkout Page

**Goal**: Create a checkout page with cart summary and payment form

**Acceptance Criteria**:
- [ ] Cart summary displays item list, quantities, and total
- [ ] Payment form includes card number, expiry, CVV with proper labels
- [ ] All form fields are keyboard-accessible
- [ ] Layout uses only DS-approved components

**Technical Notes**:
- Use `CheckoutLayout` two-column pattern
- Components: Button (primary), Input, Card, CartSummary
- Tokens: spacing.large, color.primary, typography.body
- Implement aria-live region for form error announcements

**Related Components**: Button, Input, Card
**Run ID**: <run_id>
```

### 5. Publisher Agent

**Mission**: Write or export the final artifact to the appropriate location.

**Skills**: `write_output_files`

**Rules**:
- Ask the user to confirm the output path before writing files
- If writing multiple files, list them all and get confirmation
- Add a HIVE metadata comment at the top of each generated file:
  ```
  // Generated by HIVE Deliver Hub
  // Blueprint: <blueprint.id>
  // Run ID: <run_id>
  // Date: <ISO date>
  ```
- Do not overwrite existing files without explicit confirmation

## Final Deliver Output

Write to `state.deliver`:

```json
{
  "artifacts": [
    {
      "type": "code",
      "name": "CheckoutPage.tsx",
      "path": "src/pages/CheckoutPage.tsx",
      "content": "..."
    },
    {
      "type": "specification",
      "name": "checkout-page-spec.md",
      "path": "docs/checkout-page-spec.md",
      "content": "..."
    }
  ],
  "format": "code+markdown",
  "output": {
    "primary": "CheckoutPage.tsx",
    "secondary": ["checkout-page-spec.md"]
  },
  "metadata": {
    "blueprint_id": "...",
    "run_id": "...",
    "delivered_at": "..."
  }
}
```

Update `state.meta.status` to `"delivered"`.

---

## Hub Rules

1. **Never deliver if Verify has not passed.** This is the primary rule of Deliver Hub.
2. **Deliver must be a faithful rendering of the blueprint.** No new design decisions at this stage.
3. **Accessibility fixes from Verify must be present in the code.** Check `state.verify.fixes` and apply all to the generated code.
4. **Generated code must be runnable.** It should compile and render — use TODO comments for unimplemented business logic, not pseudocode placeholders.
5. **Include the Run ID in all generated artifacts.** This enables traceability back to the HIVE state.

---

## Anti-Patterns

- Do not deliver if `state.verify.valid = false` — not even "just a preview"
- Do not make layout or component decisions in Deliver — those are Compose's decisions
- Do not produce prose narration as the output — produce structured artifacts (code, markdown, JSON)
- Do not hardcode values that should be tokens
- Do not silently ignore `state.verify.fixes` — all issues with fixes must be resolved in the generated code
